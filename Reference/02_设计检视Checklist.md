# 设计检视 Checklist

## 维度 1: 视觉一致性 (Visual Consistency)
- [ ] **色彩系统**: 是否严格遵循设计规范色板？(报警红、安全绿等语义色是否混用)
- [ ] **字体排版**: 标题、正文、辅助文字字号及颜色等级是否清晰？
- [ ] **图标使用**: 图标风格是否统一，语义是否明确？
- [ ] **间距比例**: 栅格系统是否对齐，模块间距是否遵循 8px 倍数原则？

## 维度 2: 交互效率 (Efficiency)
- [ ] **信息密度**: 列表/详情页信息排布是否合理，有无大量留白或过载？
- [ ] **核心操作**: 主操作是否明显，次要操作是否收纳得当？
- [ ] **操作深度**: 核心任务的点击次数是否最少？
- [ ] **批量操作**: 针对大数据量场景，是否提供批量处理能力？

## 维度 3: 认知与反馈 (Cognition & Feedback)
- [ ] **文案表达**: 是否使用术语而非黑话？操作反馈是否具有建设性？
- [ ] **状态反馈**: 所有的操作是否都有明确的反馈（Loading, Success, Toast, Empty）？
- [ ] **容错机制**: 危险操作是否有二次确认？报错是否能引导用户修复？

---

## 维度 4: 总览仪表盘类 (Dashboard)
- [ ] **价值定位**: 是否聚焦目标用户关键 Job (如：工程师每日运维、向上汇报)？信息呈现是否围绕叙事逻辑？
- [ ] **界面布局**: 是否采用卡片式布局？逻辑是否遵循使用场景？是否支持自定义卡片顺序/显隐？
- [ ] **信息简洁**: 层次分明，分区清晰，重点突出？是否尽量可视化呈现，避免了大篇幅文字或表格？
- [ ] **点击下钻**: 关键数据（待办、风险等）是否支持直接点击下钻查看详情？
- [ ] **数据逻辑**: 取值范围是否清晰？是否存在逻辑错误（如：子项之和不等于总数）？下钻数据是否能对齐？
- [ ] **数据时效**: 实时性数据（如拦截数）更新是否及时？(原则上实时，最大延迟不超过 2 分钟)。
- [ ] **无数据场景**: 是否有无数据场景的价值传递或情感化设计？(不可不做任何处理)。
- [ ] **加载性能**: 默认加载时长是否达标？(SaaS 产品 < 3s，硬件/盒子类 < 5s)。

## 维度 5: 运营报告类 (Operation Report)
- [ ] **价值定位**: 模版场景定位是否清晰 (运营类/汇报类)？结构是否围绕场景叙事？
- [ ] **结构设计**: 是否遵循“总-分-总”或“总-分”结构？关键信息是否在总结模块清晰呈现？
- [ ] **章节逻辑**: 信息呈现层次分明，章节逻辑清晰，重点突出？
- [ ] **色彩搭配**: 是否专业、稳重，适合长时间阅读，避免过于花哨？
- [ ] **导出灵活性**: 是否考虑了用户二次编辑场景？(如提供 Word 格式)。
- [ ] **交互合理性**: 离线文档（PDF/Word/PPT）内是否存在信息折叠或按钮交互？(除 HTML 报告外，不允许存在需要交互的操作，但支持内部跳转)。
- [ ] **数据逻辑**: 数据间逻辑是否清晰，无明显歧义或计算错误？
- [ ] **文案质量**: 措辞得当，简洁有重点？是否存在语病或错别字？
- [ ] **无数据场景**: 报告中无数据模块是否有妥善的处理或解释？

## 维度 6: 配置管理类 (Configuration Management)
- [ ] **流程极简**: 一个配置任务原则上操作 <= 3 步，复杂任务 <= 5 步？是否在当前页完成，避免页面跳出？
- [ ] **配置项清晰**: 必填项是否 <= 8 项？非必填是否提供默认值或放入高级配置？必填项是否红星标注？
- [ ] **引导提示**: 是否结合小 i、占位符进行提示？关键配置旁是否提供案例参考（如高危端口使用）？
- [ ] **操作反馈**: 反馈是否及时（加载动画、明确的成功/失败弹窗/消息提示）？
- [ ] **保存与恢复**: 复杂配置（>8项）是否提供自动保存机制？崩溃或异常丢失后能否恢复编辑内容？
- [ ] **中断恢复**: 步骤 >= 3 步的长流程是否支持随时中断并保存进度，后续继续编辑？
- [ ] **操作便捷性**: 是否提供批量操作、默认模板、或快速定位策略等交互？
- [ ] **风险二次确认**: 高危或不可逆操作（如规则删除、大规模关闭终端）是否进行二次确认？必要时是否有动态令牌验证？
- [ ] **审计记录**: 是否支持历史操作记录管理，方便回溯（谁在什么时间操作了哪个字段）？

## 维度 7: 部署类 (Deployment)
- [ ] **引导流程**: 复杂任务是否分步引导？核心 <= 3 步，复杂 <= 5 步？开始前是否有步骤预览？
- [ ] **进度反馈**: 耗时较长的部署是否提供阶段性进展展示？是否提供进度条？
- [ ] **配置项简洁**: 必填项控制在 8 项以内？是否有清晰的名称描述与红星标注？是否提供案例参考？
- [ ] **开箱即用**: 针对不同诉求客户，是否提供内置的开箱即用模板（SaaS 场景）或预设规则？
- [ ] **防错指引**: 是否实时校验格式错误？错误提示是否包含明确的修复指引（如：告知具体的数值要求）？
- [ ] **故障自闭环**: 报错信息是否具备可读性，尽可能指导一线人员自行闭环解决故障？
- [ ] **极端场景健壮性**: 长表单是否提供数据暂存？断网提交是否有友好提示？强制崩溃恢复后数据是否留存？

## 维度 8: 运营管理类 (Alerts/Assets/Events)
- [ ] **布局直观**: 筛选、统计、列表、操作区布局是否简洁？用户能否在 2s 内理解分区？屏占比是否合理？
- [ ] **信息高效**: 关键信息是否能在一屏内展示完整？用户能否快速获取核心指标？
- [ ] **减法交互**: 是否做到“非必要不跳转”？任务处理（新增、详情）是否优先使用弹窗或抽屉？
- [ ] **统计联动**: 数据统计图表是否支持点击后快速过滤列表数据？
- [ ] **筛选能力**: 重筛选场景下，表头列是否支持快速筛选？是否提供简易与复合两种筛选模式？
- [ ] **操作批量**: 是否提供批量修改状态等操作以减少冗余步骤？
- [ ] **状态记忆**: 返回后是否能记住之前的筛选条件、排序及自定义显示的字段？
- [ ] **连续性定位**: 下钻后返回列表，是否能自动定位在上次查看的数据位置？
- [ ] **通告触达**: 关键事件是否支持多通道（短信、公众号、邮件）自定义通知？
- [ ] **局部刷新**: 单条告警/事件操作完成后，是否采用局部静默刷新，避免整页重载？

## 维度 9: 工单管理类 (Ticket Management)
- [ ] **流程极简**: 发起工单是否 <= 3 步？创建流程模板是否 <= 3 步？
- [ ] **配置快捷**: 是否通过选择操作代替过多的输入？必填项是否控制在 8 项以内？
- [ ] **流转可视**: 是否可实时查看处理进度、各节点处理人、处理时间及关键动作？
- [ ] **通知灵活**: 是否支持手机、邮箱、微信等多种提醒方式，且通道可配置？
- [ ] **重点突出**: 是否能快速筛选出“待我处理”？紧急或即将超时的工单是否有醒目的颜色提醒？
- [ ] **权限严谨**: 是否支持细粒度权限控制（如：仅上级催办、仅发起人撤销），避免越权？
- [ ] **平台联动**: 工单处理状态是否自动与关联模块（如告警模块）同步，并保留处置备注？

## 维度 10: 日志分析类 (Log Analysis)
- [ ] **入口直观**: 用户能否在 1s 内找到搜索框、筛选条件等关键功能入口？
- [ ] **详情简洁**: 日志详情（如五元组）是否分类清晰？非关键信息是否默认折叠？
- [ ] **高频外露**: 时间、类型、访问方向等高频检索选项是否直接展示？
- [ ] **攻击高亮**: 数据包的关键攻击特征（如 SQL 注入、XSS）是否通过颜色或背景高亮突出？
- [ ] **快捷过滤**: 列表或详情中的字段是否支持一键加入检索过滤框？

## 维度 11: 详情展示类 (Detail Display)
- [ ] **呈现形式**: 是否优先使用抽屉呈现详情？若必须跳转，是否有面包屑/标签页快速返回？
- [ ] **层级分明**: 区域布局是否合理？信息呈现是否主次分明、模块层级清晰？
- [ ] **关键突出**: 告警级别、影响资产等关键信息是否通过颜色或加粗突出显示？
- [ ] **操作统一**: 主操作区位置是否统一？跟随字段的快捷操作分布是否规律统一？
- [ ] **连续处理**: 是否提供“上一条/下一条”切换按钮？长详情是否提供内部快捷导航？

## 维度 12: 任务下发/触发类 (Task Dispatch)
- [ ] **流程极简**: 任务创建流程是否 <= 3 步？(如：选择目标 -> 配置参数 -> 确认下发)。
- [ ] **配置项控制**: 必填项是否控制在 8 项以内？是否有清晰的名称描述与红星标注？
- [ ] **执行可视化**: 任务执行过程中，是否实时展示扫描进度 (进度条/百分比)、已完成资产数及耗时？
- [ ] **异常处理机制**: 遇到网络中断等情况是否支持自动暂停？恢复后是否可继续执行？是否提供手动重试操作？
- [ ] **结果呈现**: 任务完成后，扫描结果是否以清晰、直观的方式（如仪表盘/图表）展示？
- [ ] **高危二次确认**: 涉及中断任务、删除规则等高危操作时，是否进行二次确认？
- [ ] **日志可溯**: 任务下发日志是否清晰记录了操作人、时间、任务内容及具体动作？
- [ ] **异步通知**: 异步任务完成或异常中断时，是否通过邮件、短信或公众号及时通知用户？

## 维度 13: 报表生成类 (Report Generation)
- [ ] **入口易达**: 周期运营类报告入口在信息架构上是否处于易获得的位置？
- [ ] **模版清晰**: 内置报告模版名称是否易懂？是否提供了适用场景说明？
- [ ] **快捷导出**: 支持自定义时间范围后一键生成？高级参数是否提供合理的默认值？
- [ ] **异步管理**: 生成任务是否采用异步处理？是否提供导出历史管理界面？
- [ ] **生成反馈**: 任务执行中是否有明确的进度展示？完成后是否支持自动跳转下载或手动下载？
- [ ] **导出记录**: 导出日志是否清晰记录了谁、在什么时间、导出了哪份报告？

## 维度 14: 流程/剧本编排类 (SOAR/Orchestration)
- [ ] **分层引导**: 复杂剧本创建是否拆分为分阶段配置（如：基本设置 -> 流程设计 -> 动作编排）？每阶段是否 <= 3 步？
- [ ] **草稿暂存**: 是否支持保存未完成的流程为草稿？恢复编辑时能否自动定位到上次编辑位置？
- [ ] **预演测试**: 在正式启用预案前，是否支持用户进行模拟测试，并根据结果调整逻辑？
- [ ] **流程可视**: 执行流程是否清晰可视？用户能否通过详情页快速定位到当前步骤及报错节点？
- [ ] **追溯严谨**: 执行记录是否完整记录了执行时间、触发条件、动作及结果？是否可查询且无漏报？
- [ ] **设备联动反馈**: 联动设备的对接状态是否实时可见？联动异常时是否有明确提示？
- [ ] **错误恢复引导**: 失败时是否展示用户可理解的原因（非技术术语）？是否给出具体的修复建议？
- [ ] **资源占用**: 自动化任务执行时，CPU/内存占用率是否合理 (建议 <= 15%)？任务积压时是否有提示？

## 维度 15: 拓扑编辑可视类 (Topology Editing)
- [ ] **布局合理性**: 工具栏（顶部）、属性面板（右侧）及显示区域划分是否明晰，避免视觉疲劳？
- [ ] **信息清晰度**: 设备名称、IP、连接状态等关键信息是否清晰？连接关系是否可见并避免线条交叉重叠？
- [ ] **交互友好性**: 是否支持平滑的缩放、平移、拖拽等基础交互操作？
- [ ] **加载性能**: 包含 100 个设备的拓扑图是否能在 3s 内加载完成？
- [ ] **响应速度**: 画布上的交互操作（如拖拽设备）系统是否能在 1s 内给出响应并更新？
- [ ] **风险二次确认**: 画布上的高危操作是否具备二次确认逻辑？

## 维度 16: 数据流转可视类 (Data Flow)
- [ ] **视线流清晰**: 数据流向的方向性是否清晰可视？是否将流转的不同阶段进行分区展示？连接线是否整洁？
- [ ] **语义识别**: 对象模块名称、逻辑、元素识别是否易读？是否使用了统一的表达元素（如应用图标、敏感颜色）？
- [ ] **异常定位**: 异常点与关键节点（如外发拒绝、告警）是否有突出的视觉元素方便快速定位？
- [ ] **操作流畅性**: 选中高亮、展开收起、联动过滤筛选等交互是否流畅且符合预期？
- [ ] **配置记忆**: 系统能否记住用户的上一次操作（如视图配置、时间筛选、画布收起状态）？

## 维度 17: 反馈类 (Modal/Tips/Toast/Announcement)
- [ ] **形式合理性**: 不同场景选择方式是否得当？(模态弹窗：需告知并操作；Toast：弱反馈，不打断操作；Tips：辅助提示)。
- [ ] **文案语义化**: 重点清晰，简洁明了？是否符合用户语言，避免直接呈现后端错误代码？
- [ ] **指引建设性**: 错误提示是否明确告知了下一步如何解决？(特别是针对用户任务操作类场景)。
- [ ] **交互便捷性**: 提示框内是否有明确的下一步动作按钮，方便用户一键进入任务？
- [ ] **强提示规范**: 执行高危/危险操作时，是否强制采用模态强提示，并视情况增加验证机制？
